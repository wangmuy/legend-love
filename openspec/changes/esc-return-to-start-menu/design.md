## Context

当前事件驱动版游戏中，重新开始流程如下：
1. 开始菜单显示（重新开始/载入进度/离开游戏）
2. 选择"重新开始"进入属性选择界面
3. 属性选择界面显示随机生成的属性
4. 用户选择"是"确认属性，或"否"重新随机

问题在于：用户在属性选择界面无法通过ESC键返回到开始菜单。这在原版游戏中也不支持，但用户期望有此功能。

当前架构使用协程和状态机管理流程，`startNewGame` 是一个协程，调用 `MenuAsync.ShowMenu2Coroutine` 显示属性选择菜单。

## Goals / Non-Goals

**Goals:**
- 在属性选择界面支持ESC键返回到开始菜单
- 保持事件驱动架构，使用协程和状态机
- 不破坏现有的菜单系统和其他功能

**Non-Goals:**
- 修改原版jymain.lua中的同步代码
- 添加除ESC键之外的其他导航方式
- 修改属性选择的基本逻辑（只是添加返回功能）

## Decisions

### 1. 使用特殊返回值表示ESC返回

**方案**: 当用户按ESC键时，`MenuAsync.ShowMenu2Coroutine` 返回特殊值（如-1或"esc"），表示用户取消操作。

**理由**: 
- 最小侵入性修改，不需要大幅重构菜单系统
- 与现有返回值机制兼容
- 易于在调用方处理

**替代方案**: 
- 抛出异常：过于重量级，不符合Lua习惯
- 修改菜单状态机：改动太大，影响其他菜单

### 2. 在startNewGame中处理返回逻辑

**方案**: 在 `startNewGame` 协程中检查菜单返回值，如果为ESC特殊值，则通过 `EventBridge.switchState` 返回到开始菜单状态。

**理由**:
- 符合事件驱动架构，使用状态机切换
- 保持协程的清晰流程
- 易于扩展其他返回点

### 3. 修改MenuStateMachine支持ESC返回

**方案**: 修改 `MenuStateMachine:handleInput`，当检测到ESC键且 `isEsc == 1` 时，关闭菜单并返回特殊值。

**理由**:
- 复用现有的ESC处理机制
- 只需要修改 `isEsc` 参数从0到1
- 与原版ShowMenu2行为一致

## Risks / Trade-offs

**风险**: 修改菜单系统可能影响其他使用 `ShowMenu2` 的地方
- **缓解**: 仔细测试所有使用横向菜单的场景，确保ESC行为正确

**风险**: 协程中状态切换可能导致状态不一致
- **缓解**: 在切换状态前清理资源（如清除绘制回调），确保干净退出

**权衡**: 使用特殊返回值 vs 修改菜单数据结构
- 选择特殊返回值，因为改动最小，不影响其他功能
