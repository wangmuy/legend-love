## Context

当前金庸群侠传Love2D复刻版使用自定义的游戏循环架构:
- `main.lua`中的`love.run()`被完全重写，直接调用`JY_Main()`
- `jymain.lua`中的`Game_Cycle()`使用`while`循环和`lib.GetKey()`阻塞式输入
- 渲染和逻辑在同一个循环中执行，通过`lib.Delay()`控制帧率

这种架构虽然工作正常，但没有充分利用Love2D的事件驱动特性，代码耦合度高，难以维护和扩展。

## Goals / Non-Goals

**Goals:**
- 将游戏架构改为标准Love2D事件驱动模式(love.update/love.draw)
- 将输入处理改为事件驱动(love.keypressed/keyreleased)
- 分离游戏逻辑更新和渲染逻辑
- 保持游戏效果和行为与重构前完全一致
- 不修改`script/`目录下的任何代码

**Non-Goals:**
- 不修改游戏逻辑和规则
- 不修改事件脚本(oldevent/和newevent/)
- 不添加新功能
- 不修改游戏数据和资源文件

## Decisions

### 1. 使用状态机管理游戏状态
**决策**: 创建`GameStateManager`模块管理GAME_MMAP、GAME_SMAP等状态
**理由**: 
- 清晰分离不同游戏状态的处理逻辑
- 便于状态切换和初始化/清理
- 与Love2D事件回调良好配合

**替代方案**: 保留全局变量JY.Status，在update中switch-case处理
**未选择原因**: 状态机更清晰，便于扩展

### 2. 输入事件队列
**决策**: 使用队列缓存键盘事件，在update中处理
**理由**:
- Love2D事件回调与游戏逻辑更新可能不同步
- 队列确保输入事件按顺序处理
- 兼容原有的按键处理逻辑

### 3. 保留原有游戏循环语义
**决策**: `Game_Cycle()`中的逻辑拆分为update和draw两部分，但保持每帧执行顺序
**理由**:
- 确保游戏行为一致
- 最小化对jymain.lua的修改
- 原有定时逻辑(JY.MyTick等)继续工作

### 4. 渲染调用点调整
**决策**: 将`ShowScreen()`和`lib.DrawMMap()`等渲染调用移到`love.draw()`
**理由**:
- Love2D标准做法
- 避免在update中直接渲染
- 支持vsync和帧率控制

## Risks / Trade-offs

**[风险] 输入延迟变化**
原架构中`lib.GetKey()`是阻塞式，新架构是事件驱动
**缓解**: 确保事件队列及时处理，保持每帧检查输入

**[风险] 渲染时序变化**
原架构在逻辑中穿插渲染，新架构分离后可能影响视觉效果
**缓解**: 仔细保持渲染调用顺序和参数

**[风险] 脚本兼容性**
事件脚本可能依赖特定的执行时序
**缓解**: 不修改script/目录，保持原有API行为

**[风险] 性能变化**
事件驱动架构可能引入额外开销
**缓解**: 预期性能持平或略有提升，需测试验证

## Migration Plan

1. **Phase 1**: 创建新的状态机和事件处理模块
2. **Phase 2**: 重构main.lua，实现标准Love2D回调
3. **Phase 3**: 修改jymain.lua中的Game_Cycle，拆分为update/draw
4. **Phase 4**: 修改lib_love.lua中的输入处理
5. **Phase 5**: 全面测试，确保行为一致

**回滚策略**: 保留原代码备份，如出现问题可快速回退

## 测试策略

### 单元测试
- **状态机测试**: 测试状态注册、切换、初始化和清理功能
- **输入管理器测试**: 测试事件队列、按键状态查询、事件处理
- **事件桥接测试**: 测试新旧API的兼容性和行为一致性

### 整体测试
- **手动测试**: 由于游戏涉及复杂的UI交互和场景切换，整体功能测试需要用户手动进行
- **日志调试**: 在关键路径添加详细日志，通过`lib.Debug()`输出到debug.txt
- **用户反馈**: 用户测试时记录问题场景，通过日志文件分析问题原因
- **对比验证**: 对比重构前后游戏行为，确保视觉效果和操作体验一致
