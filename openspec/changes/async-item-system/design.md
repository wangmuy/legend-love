## Context

当前物品系统基于同步阻塞调用设计，主要函数包括：
- `SelectThing`: 显示物品选择界面，使用 `while true` 循环和 `WaitKey` 处理输入
- `UseThing`: 根据物品类型调用不同的使用函数
- `UseThing_Type0` 到 `UseThing_Type4`: 分别处理剧情物品、装备、秘籍、药品、暗器
- `Menu_Thing`: 物品菜单入口，调用 `SelectThing` 和 `UseThing`

在事件驱动架构下，这些同步函数会阻塞协程执行，导致菜单无法正常显示和响应。

## Goals / Non-Goals

**Goals:**
- 实现异步版本的物品选择菜单，支持在协程中正常运行
- 重构所有物品使用逻辑为协程版本
- 保持与原有物品数据的兼容性
- 支持在小场景菜单中正常使用物品功能

**Non-Goals:**
- 不修改物品数据结构（`JY.Thing`、`JY.Base` 等）
- 不修改物品图片和显示资源
- 不新增物品类型或功能
- 不修改战斗系统中的物品使用（后续迭代处理）

## Decisions

### 1. 使用协程包装模式
**决策**: 将同步函数包装为协程版本，而非完全重写。
**理由**: 
- 原有逻辑经过验证，重写风险高
- 通过 `AsyncGlobals` 机制可以透明替换同步调用
- 保持代码可维护性

**替代方案**: 完全重写为回调模式（ rejected：代码改动量大，易引入bug）

### 2. 分页显示物品列表
**决策**: 物品选择菜单使用分页显示，每页显示固定数量（如10个）。
**理由**:
- 避免一次显示过多物品导致界面拥挤
- 与原有 `SelectThing` 的双栏显示保持一致
- 简化异步菜单的实现

### 3. 物品使用流程简化
**决策**: 装备物品时先选择物品，再选择使用对象（队友）。
**理由**:
- 与原有流程一致，用户学习成本低
- 每个步骤使用独立的异步菜单
- 便于错误处理和取消操作

### 4. 错误处理策略
**决策**: 物品使用失败时显示异步消息框，不抛出异常。
**理由**:
- 保持游戏流畅性
- 用户友好（如"此人不适合配备此物品"）
- 避免协程因错误而终止

## Risks / Trade-offs

**[Risk] 原有代码耦合度高** → **Mitigation**: 逐步重构，先实现核心功能，再处理边界情况

**[Risk] 异步转换可能改变用户体验** → **Mitigation**: 保持界面布局和操作流程与原版一致

**[Risk] 战斗系统物品使用未同步修改** → **Mitigation**: 本次change仅处理小场景菜单，战斗系统单独处理

**[Trade-off] 代码复杂度增加** → 需要维护同步和异步两套逻辑，直到完全迁移完成

## Migration Plan

1. **Phase 1**: 实现 `SelectThingAsync` 基础功能
   - 创建 `src/item_async.lua` 模块
   - 实现分页显示物品列表
   - 支持物品选择和返回

2. **Phase 2**: 实现物品使用异步化
   - 重构 `UseThing_Type1`（装备）为协程版本
   - 重构 `UseThing_Type3`（药品）为协程版本
   - 其他类型暂时显示"暂不支持"

3. **Phase 3**: 集成到菜单系统
   - 更新 `JyMainAsync.Menu_Thing` 调用新的异步函数
   - 测试物品选择和使用流程

4. **Phase 4**: 完整功能实现
   - 实现所有物品类型的异步版本
   - 处理剧情物品、秘籍、暗器等特殊逻辑
